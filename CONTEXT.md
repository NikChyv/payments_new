## 1. Что за проект и зачем

Бухгалтерская аутсорсинговая компания, 50+ клиентов. Большинству клиентов нужно регулярно
делать платежи.

**Проблема, которую решаем:**
- Клиенты присылают поручения на оплату в разных каналах (Telegram, Viber, звонки) и в
  разных форматах (текст, фото счёта, накладная), часто без дат и деталей.
- Нет CRM и единого списка — платежи теряются в переписке, «просматриваются и забываются».
- Срываются сроки. Был показательный случай: поручение пришло в пятницу к концу дня,
  сообщение просто просмотрели, платёж не сделали — в выходные у клиента встала реклама.

**Как устроен процесс:** бухгалтер сам проводит платёж через банк-клиент клиента и
отправляет чек об операции.

**Цель:** уйти от «забывания и стикеров на мониторе» — структурированный приём заявок +
единая очередь по срокам. Дополнительно — позиционирование «у нас платежи не теряются,
всё автоматизировано».

## 2. Что сделали — решение

Одно веб-приложение (**один файл** `index.html`, в чате назывался `platezhi.html`),
два экрана:

**Экран клиента (форма заявки):** обязательные поля — компания, получатель, сумма,
реквизиты/ИНН, дата платежа, периодичность (разовый / еженедельно / ежемесячно),
назначение, файл (счёт/фото), нужен ли чек. Клиент физически не может прислать
«фото без пояснений».

**Экран бухгалтера (очередь):** единый список вместо стикеров.
- Счётчики: Просрочено / На сегодня / Ближайшие 7 дней / В работе; красный баннер, если горит.
- Сортировка по сроку, цветная подсветка (красный — просрочено, оранжевый — сегодня).
- Статусы в один клик: Новая → В работе → Оплачено → Чек отправлен.
- Регулярные платежи **сами создают следующий** после отметки «оплачено».
- Поиск и фильтр по клиенту. Прикреплённый файл открывается по клику на «📎».

## 3. Архитектура / стек (важно понимать роли)

- **GitHub Pages** — хостит страницу и даёт публичную ссылку. *(Это НЕ база данных.)*
- **Supabase** (бесплатный тариф) — база данных (таблица `payments`) + хранилище файлов
  (бакет `files`) + автоматический REST API.
- **Офлайн-фолбэк:** пока ключи Supabase не вписаны, приложение работает локально через
  `localStorage` с демо-данными — удобно проверять интерфейс без настройки.
- Supabase подключается через CDN-клиент, без сборки проекта (чистый HTML+JS в одном файле).

## 5. Ключевые детали конфигурации

**Где вписаны ключи** — вверху блока `<script>`, секция `НАСТРОЙКА SUPABASE`:

```js
var SUPABASE_URL = "https://<ref>.supabase.co"; // ВАЖНО: без /rest/v1 в конце!
var SUPABASE_KEY = "<anon public key>";          // ключ anon public (начинается с eyJ...)
var TABLE  = "payments";
var BUCKET = "files";
```

**SQL для таблицы** (выполнялся в Supabase → SQL Editor):

```sql
create table if not exists payments (
  id text primary key,
  client text,
  payee text,
  amount numeric,
  requisites text,
  due date,
  recurrence text,
  purpose text,
  status text,
  need_receipt boolean,
  file_url text,
  file_name text,
  created_at timestamptz default now()
);
alter table payments disable row level security;
```

**Политика Storage** (бакет `files`): операция **INSERT**, определение `bucket_id = 'files'`.
Бакет Public, поэтому чтение файлов идёт через `getPublicUrl` без отдельной политики.

**Ссылка клиента:** `БАЗОВЫЙ_АДРЕС/?client=ООО Ромашка` — открывает только форму,
компания уже подставлена и заблокирована. У каждого клиента своя ссылка.

## 6. Грабли, которые уже прошли (не повторять)

- В `SUPABASE_URL` НЕ должно быть `/rest/v1` — иначе путь задваивается (ошибка `PGRST125`, 404).
- При создании таблицы в диалоге выбирать **«Run without RLS»** (иначе anon заблокирован).
- После правки файла обновлять страницу через **Ctrl+F5** (иначе висит старый кэш).
- 404 на `/payments` = таблицы нет либо она не в схеме `public`.
- Двойной клик по файлу = открыть в браузере (запуск). Чтобы править код — открывать
  текстовым редактором (Блокнот / VS Code), а не браузером.

## 7. Известные ограничения текущего прототипа

- RLS отключён → любой со ссылкой и ключом может писать в базу. Ок для приватного теста,
  НЕ для реальных клиентов.
- Напоминания пока **визуальные** (работают, когда экран открыт). Push-уведомлений нет.
- Режим рассчитан на простую пробу: 1 бухгалтер + 1 клиент.
- Периодичность «ежемесячно» при дате 31-го числа может слегка съезжать (мелкий edge-case).

## 8. Что делать дальше (роадмап)

1. **Уведомления бухгалтеру в Telegram** при новой заявке и при наступлении срока — главный
   шаг против забывания, чтобы не зависеть от «открыт ли экран».
2. **Включить RLS + вход** для клиентов и бухгалтеров (безопасность финансовых и
   персональных данных, 152-ФЗ).
3. **Мультипользовательский режим** (несколько бухгалтеров, роли).
4. **Генератор ссылок** на экране бухгалтера: ввёл название компании → готовая ссылка с
   кнопкой «Скопировать» (чтобы не кодировать вручную).
5. *(Опционально)* **AI-распознавание** фото/накладной: автоматически вытаскивать
   получателя, ИНН, сумму, счёт — клиент только подтверждает.
6. Когда проект перерастёт один файл (вход, бот, серверный код) — переносить в **Claude Code**.
